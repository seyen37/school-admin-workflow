# 測試流程與驗收標準

本文件給「安裝完想驗證有沒有壞」的人。第一次安裝請依序跑完 1–4 項，其餘是進階測試。

每項測試都有明確的 **✅ 通過條件**、**❌ 失敗時看哪裡**、**🔁 回滾步驟**。

---

## 1. 測試 Drive 連線

**執行：** Apps Script 編輯器選函式 `testDriveFolder` → 執行

**做什麼：** 嘗試讀取 `CONFIG.ROOT_FOLDER_ID` 並印出資料夾名稱。

- ✅ **通過：** Logger 顯示你的總資料夾名稱（例：`學校行政專案工作流`）
- ❌ **失敗：**
  - 訊息「找不到 ID」→ 檢查第 2 步抄的 ID 是否完整、有沒有多餘空白
  - 訊息「沒有權限」→ 檢查你登入的 Google 帳號是不是該資料夾的擁有者或編輯者
- 🔁 **回滾：** 不需要，這個測試沒有寫入

---

## 2. 測試 setup 建立基礎結構

**前提：** 第一次安裝、總控表還沒建立。

**執行：** `setupAdminWorkflow` → 執行

- ✅ **通過：**
  - Logger 顯示「安裝完成」+ 兩個表單網址 + 總控表網址
  - Drive 總資料夾下出現五個子資料夾：`01_公文專案啟動表`、`02_專案總控表`、`03_專案資料夾`、`04_專案階段日期新增表`、`06_Apps Script`
  - 觸發器頁面（Apps Script 左側「觸發條件」）出現兩個觸發器：`onFormSubmit`、`onMilestoneFormSubmit`
- ❌ **失敗：**
  - 「找不到 CALENDAR_ID」→ 改 `CONFIG.CALENDAR_ID` 為 `primary` 試試
  - 「Gmail send 權限不足」→ 重新跑授權對話框，這次同意 Gmail 權限
  - 觸發器沒建立 → 看 `installFormSubmitTrigger` 的錯誤 log
- 🔁 **回滾：** 跑 `resetAdminWorkflow(true)`（dry-run）看會刪什麼，再決定要不要真執行 `resetAdminWorkflow(false)`

---

## 3. 測試新建專案（核心驗證）

**執行：** 打開「公文專案啟動表」填寫網址，填一筆測試資料：

```
專案年度：測試
承辦處室：資訊組
專案名稱：第一次安裝驗證
承辦人：（你的名字）
承辦人Email：（你的 Email）
活動日期：（任選未來日期）
成果繳交期限：（活動日後一週）
經費核銷期限：（活動日後兩週）
其餘欄位空白即可
```

送出後 **5 秒內** 檢查：

- ✅ **通過（全部勾才算）：**
  - [ ] 總控表新增一列，ProjectCode 為 `測試-資訊組-001`
  - [ ] `03_專案資料夾` 下出現 `測試_資訊組_第一次安裝驗證` 資料夾
  - [ ] 該資料夾內有 **11 個子資料夾**（00 ~ 99）
  - [ ] `99_系統產生文件` 內有 3 個檔案：專案紀錄 Docs、待辦追蹤表、成果檢核表
  - [ ] 待辦追蹤表內有 **10 列預設任務**
  - [ ] 成果檢核表內有 **10 列預設檢核項目**
  - [ ] Calendar 出現 3 個事件（活動日、成果期限、經費期限）+ 2 個提醒事件（成果前 7 天、經費前 7 天）
  - [ ] 你的 Email 收到一封「【行政專案已建立】第一次安裝驗證」
- ❌ **失敗：**
  - 沒收到通知信 → 看 Apps Script 執行作業，找 `onFormSubmit` 的錯誤
  - 資料夾沒建 → 觸發器可能沒裝好，回 2 重跑 `setupAdminWorkflow`
  - 總控表沒新列 → 看執行 log 是否有「Range out of bounds」（欄位被改動）
- 🔁 **回滾：**
  1. 刪掉 `03_專案資料夾/測試_資訊組_第一次安裝驗證/`
  2. 從總控表刪掉該列
  3. 從 Calendar 刪掉那 5 個事件
  4. 從 PropertiesService 不需要動

---

## 4. 測試新增階段日期

**前提：** 已完成測試 3，總控表內有 `測試-資訊組-001`。

**執行：**
1. 從總控表複製 ProjectCode：`測試-資訊組-001`
2. 打開「專案階段日期新增表」
3. 填：
   ```
   專案編號：測試-資訊組-001
   專案名稱：第一次安裝驗證
   日期類型：報名截止日
   日期：（任選未來日期）
   提醒設定：✅前7天 ✅前3天 ✅當天
   負責人：（你的名字）
   負責人Email：（你的 Email）
   是否寫入待辦追蹤表：是
   是否建立Calendar提醒：是
   ```
4. 送出

- ✅ **通過：**
  - [ ] 「階段日期紀錄」工作表新增一列，狀態為「有效」
  - [ ] 該專案的待辦追蹤表新增一列，任務名稱為「【報名截止日】第一次安裝驗證」，階段為「籌備」
  - [ ] Calendar 出現一個事件 `【報名截止日】第一次安裝驗證`
  - [ ] 該 Calendar 事件**有 3 個提醒**（你可以點開事件看右側）
  - [ ] 負責人 Email 收到通知
- ❌ **失敗：**
  - 「階段日期紀錄」狀態為「錯誤」→ 看備註欄的錯誤訊息
  - Calendar 沒提醒 → 表示彈跳提醒沒設好，看 `lib/calendar.gs`
- 🔁 **回滾：** 刪掉新增的 Calendar 事件 + 階段日期紀錄那一列 + 待辦表那一列

---

## 5. 測試 Calendar 彈跳提醒（重要！）

> 這條是本 fork 對原作的**關鍵改進**——確認 popup reminder 真的會跳。

**執行：** 在「專案階段日期新增表」填一筆，日期設為**今天**，提醒設「當天提醒」。

- ✅ **通過：** Calendar 事件詳情顯示「09:00 通知 — 彈出式」
- ❌ **失敗：** 只看到「全天」沒有提醒設定 → `lib/calendar.gs` 的 `addPopupReminder` 沒呼叫到
- 🔁 **回滾：** 刪掉事件

---

## 6. 測試 dedupe（防止重複送出）

**執行：** 重複用測試 3 的同樣資料**再填一次**送出。

- ✅ **通過：**
  - 總控表**不會**多一列
  - Drive **不會**多一個資料夾
  - 你會收到一封「已存在相同專案，未重複建立」通知信
- ❌ **失敗：**
  - 又建了第二份 → `lib/utils.gs` 的 `findExistingProject` 沒運作
- 🔁 **回滾：** 如果重複建了，按照測試 3 的回滾流程刪掉重複的那份

---

## 7. 測試錯誤處理

**執行：** 在「專案階段日期新增表」填**不存在的專案編號**（例：`999-不存在-001`）送出。

- ✅ **通過：**
  - 「階段日期紀錄」新增一列，狀態為「錯誤」
  - 待辦表**沒有**新增任務
  - Calendar **沒有**建立事件
  - 你（填表人）或 Admin 收到錯誤通知信
- ❌ **失敗：**
  - 沒人收到通知 → 看 `notifyAdminError` 是否被呼叫
  - 待辦表還是被寫入 → `findProjectByCode` 的 null 處理有 bug

---

## 8. 測試 Throttle（重複錯誤不灌爆信箱）

**執行：** 連續送出 5 筆都用不存在的專案編號。

- ✅ **通過：**
  - 第一封錯誤通知會收到
  - **第二到第五封不會收到**（10 分鐘內相同錯誤被 throttle）
  - 「階段日期紀錄」仍會新增 5 列（資料完整保留，只是通知被合併）

---

## 9. 測試多日曆派發（如有設定 `CALENDARS`）

**前提：** 已在 CONFIG 設定多個日曆，例如 `教務處`、`學務處` 對應不同 Calendar。

**執行：** 各填一筆指定不同處室的公文專案啟動表。

- ✅ **通過：** 事件依承辦處室出現在對應的 Calendar
- ❌ **失敗：** 全跑到 `default` → `lib/calendar.gs` 的 `getCalendarFor(office)` 沒從 CONFIG 讀

---

## 10. 測試 Reset Dry-run

**執行：** 跑 `resetAdminWorkflow(true)`（dry-run）。

- ✅ **通過：** Logger 列出所有「將會」刪除的資源，但**沒有任何東西真的被刪**
- ❌ **失敗：** 東西被刪了 → 立刻停止，回 [05-uninstall.md](./05-uninstall.md) 看手動恢復

---

## 整合測試（給上線前的學校跑）

完整跑一個假專案，從建專案 → 加 5 個階段日期 → 寫成果報告 → 結案，看一週後 Calendar 提醒是否準時跳。

通常需要 7–10 天，建議在實際辦活動前一個月先跑。
