## 決策 1：Fork vs 從零開始

**情境**：使用者已參考 mihozip/google-workspace-admin-project-workflow，覺得概念好但想做大幅改寫。是直接 fork 後改、還是視原作為靈感、從零自寫？

**選項**：
- A. **Fork + 大幅改寫，保留原作致謝**（README 顯眼處 + ACKNOWLEDGEMENTS.md + LICENSE 雙重 copyright）
- B. 全新原創，僅在 README 底部小區塊提及原作
- C. 完全不提原作

**決定**：選 A。

**考量**：
1. 原作 mihozip 用 MIT License，**法律上允許 fork + 修改，但要求保留版權聲明**。A 是最合規路徑。
2. **在學校場域，學術倫理與透明度的價值大於「自我代表作」的虛榮**。B/C 容易被質疑「為什麼跟原作那麼像」。
3. A 也最尊重原作的開創性貢獻——「公文 → 自動建專案 → 追蹤 → 成果」這條工作流概念、11 個資料夾命名、雙表單架構都是原作智慧財產。
4. 風險：A 的「我也參與設計」程度可能被低估。**緩解：在 ACKNOWLEDGEMENTS.md 詳列本 fork 的擴充範圍**（文件層 / 程式層 / 教材 / 安全），讓貢獻分工清晰。

**教訓**：
- **「致謝強度」是設計選擇，不是法律最小要求**。MIT License 只要求保留 copyright notice；但選擇「顯眼三層致謝」是策略——把學術倫理變成「不會被質疑」的護城河。
- **fork 不是恥辱**，許多優秀開源專案都是 fork 後改進（Linux 多個發行版、Chromium 與 Edge / Brave 等）。重點是「fork 後做了什麼」，而不是「有沒有從頭寫」。

## 決策 2：新 repo 名稱

**情境**：原作叫 `google-workspace-admin-project-workflow`，名稱長且技術導向。本 fork 主打學校場域、目標讀者是主任/組長，名稱要重新取。

**選項**：
- A. 沿用原名 `google-workspace-admin-project-workflow`
- B. **`school-admin-workflow`**（白話、好記）
- C. `google-workspace-admin-starter`（強調 starter 性質）
- D. 中文名 `公文專案啟動器`（GitHub 接受但慣例是英文）

**決定**：選 B。

**考量**：
1. A 過長、技術詞太多，主任搜尋時不直覺
2. **B 三個英文單字、語意明確、SEO 友善**，符合「主任/組長為主、IT 為輔」的讀者定位
3. C 強調「starter」但本 fork 已是完整工具，starter 反而貶值
4. D 中文 repo 名 GitHub 支援但工程圈不慣，跨國協作不便
5. **本機資料夾名沿用 `admin-project-workflow`**，GitHub repo 名與本地不一致沒關係——這是 Git 設計上允許的。

**教訓**：
- **本機資料夾名 ≠ GitHub repo 名**。命名時可以分開考慮，本機自由、GitHub 精緻。
- **白話勝過技術精準**：讀者寫的搜尋字詞是「school admin workflow」，不是「google workspace administrative project workflow」。

## 決策 3：License 選擇

**情境**：原作 MIT License。本 fork 是否沿用？還是換成更嚴格的 CC BY-SA、或保留版權自定？

**選項**：
- A. **沿用 MIT**，保留原作版權 + 加上本 fork 維護者版權
- B. 升級為 CC BY-SA 4.0（要求衍生作品也要 BY-SA）
- C. 改為 AGPL（最嚴格的 copyleft）
- D. 自定版權（不開源）

**決定**：選 A。

**考量**：
1. **MIT License 允許衍生作品換 License**，但這違反開源社群慣例。沿用 MIT 是最尊重原作的做法。
2. B 看似保護衍生鏈但實際對學校場域沒幫助——學校老師 fork 後想加自己 metadata，要被 BY-SA 綁住反而麻煩。
3. C 過於嚴格，會阻止學校與商業 EdTech 合作的可能性。
4. D 不開源等於違背 fork 的初衷。
5. **沿用 personal-playbook §二「智慧財產權三件套」原則**：身份鏈在 LICENSE / README / ACKNOWLEDGEMENTS 三處同步，多點互相印證。

**教訓**：
- **License 選擇有路徑依賴**：原作什麼 License，fork 通常該沿用。除非有充分理由換。
- **「身份鏈」（identity chain）的核心是多點互相印證**：法律基礎（LICENSE）+ 讀者第一印象（README）+ 顯眼致謝（ACKNOWLEDGEMENTS）三處一致，任何質疑都能找到三個獨立證據。

## 決策 4：目標讀者定位

**情境**：原作 README 同時服務「一般教師（教學工作流）」與「行政承辦人（系統工作流）」兩條主線。fork 後該維持還是聚焦？

**選項**：
- A. 維持雙主線（教學 + 系統）
- B. **聚焦單主線「行政專案啟動器」**，教學降為輔助子目錄
- C. 拆兩個 repo（系統 / 教材各一個）

**決定**：選 B。

**考量**：
1. A 維持雙主線會導致 README 視角分裂，主任看完還是不確定該看哪段
2. **B 把定位收口在「主任 / 組長 / 行政承辦人」**，README 開頭的痛點 hook（公文在信箱、附件在下載…）才有說服力
3. C 拆兩 repo 過度工程，主任不會跨 repo 找東西
4. 教材主軸仍保留在 `lessons/`，但不在 README 主舞台

**教訓**：
- **定位聚焦勝過內容完整**：當目標讀者已明確，砍掉次要內容讓主線更銳利，比保留「以備不時之需」的章節更有效。
- **fork 是重新定位的機會**：原作把所有功能塞同個 repo，fork 是合理的「重新切角度」時機。

## 決策 5：致謝策略（三層 vs 一層）

**情境**：MIT License 法律上只要求保留 copyright notice。學校場域的學術倫理更高，致謝該做到多深？

**選項**：
- A. 只保留 LICENSE 內 copyright notice（法律最小要求）
- B. LICENSE + README 底部一行
- C. **三層致謝：LICENSE 雙 copyright + README 顯眼段 + 獨立 ACKNOWLEDGEMENTS.md 詳列改寫範圍**

**決定**：選 C。

**考量**：
1. A 法律 OK 但學校場域容易被質疑「為什麼那麼像原作」
2. B 比 A 好但 ACKNOWLEDGEMENTS 仍嫌弱
3. **C 把致謝變成不會被質疑的護城河**：三處互相印證、任何 reviewer 都能立刻找到原作出處與貢獻分工
4. C 也對應 personal-playbook §二「身份鏈三件套」原則（用在原作與 fork 維護者**雙方**身上）

**教訓**：
- **學術倫理在學校場域不是「加分項」、是「基本盤」**：被檢舉抄襲的成本遠大於三層致謝的書寫成本。
- **致謝強度可以分層設計**：法律層（LICENSE）、文化層（README）、詳實層（ACKNOWLEDGEMENTS），不是「有沒有提」二元問題。
