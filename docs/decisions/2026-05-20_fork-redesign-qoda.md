# 2026-05-20：Fork 重新設計的 QODA 三輪

> 範圍：從「使用者說 OK 重新設計」到「P1/P2/P3 動手」之間，依 personal-playbook §5.9 QODA 協定走的三輪決策對齊。
>
> 起點：使用者請求「依規劃重新設計新 repo」，需釐清結構主軸、目錄結構、執行節奏才能動手。
> 終點：使用者 sign-off「O2 + 推薦的 repo 名 + 分四階段」，P1 開始執行。
>
> 對應 commits：
> - `<未推>` init: fork & rewrite from mihozip/google-workspace-admin-project-workflow

---

## 決策 1（QODA 第一輪）：倉庫結構主軸

**情境**：原作 README 同時服務教學工作流與系統工作流兩條主線。fork 後該維持兩條主線、聚焦單主線、還是拆多 repo？

**選項**：
- A. **原版兩條主線並存**：教學主線 + 系統主線
- B. **聚焦「行政專案啟動器」單主線**：教學降為 lessons/ 子目錄
- C. **多 repo 分拆**：系統 / 教學 / 範例 各一個

**決定**：選 B。

**考量**：
1. A 維持原狀，但 README 視角分裂，主任看完不知道該看哪段——這正是原作優化建議裡指出的「定位導向 vs 痛點導向」問題
2. **B 把目標讀者收口在「主任/組長/承辦人」**，README 開頭的痛點 hook（公文散落、附件難找）才有說服力
3. C 拆兩 repo 過度工程：學校主任不會跨 repo 找東西，推廣難度倍增；維護成本翻倍
4. B 也最符合 personal-playbook §一「定位聚焦勝過內容完整」原則

**教訓**：
- **fork 是重新定位的合法機會**：原作把所有功能塞同個 repo，fork 時可以合理「重新切角度」
- **「給誰看」決定文件結構**：先回答「主要讀者是誰」，再決定要保留 / 砍除哪些內容

## 決策 2（QODA 第二輪）：目錄結構與檔案命名

**情境**：B 方案確定後，新 repo 該長什麼樣？目錄怎麼編號？檔案怎麼命名？

**選項**（呈現為完整目錄樹比較）：
- A. **flat 結構**：所有檔案都在 root（README、Code.gs、Code.html…）
- B. **分層結構但無編號**：`docs/quickstart.md`、`docs/architecture.md`…
- C. **分層 + 數字前綴排序**：`docs/00-quickstart.md`、`docs/01-architecture.md`…，src/ 拆 lib/ 多模組
- D. **C + 額外 examples/ + lessons/ + scripts/**

**決定**：選 D。

**考量**：
1. A 不適合多檔案 repo，GitHub 首頁看起來會雜亂
2. B 比 A 好，但 docs/ 內的閱讀順序得靠檔名排序（quickstart 應該擺第一），需要數字前綴強制順序
3. **C 用 00-07 編號決定閱讀順序**：GitHub 預設按字母排，加數字確保「給主任先看 00-quickstart、IT 看 01-architecture」的順序對
4. **D 補三個子目錄**：lessons/ 給講師、examples/ 給潛在使用者、scripts/ 給輔助工具
5. src/ 拆 lib/ 多模組是 P3 程式碼可維護性的基礎（對應原作審查建議的「setupAdminWorkflow 拆函式」）

**教訓**：
- **檔名排序是 UX**：GitHub 首頁、Drive 資料夾、IDE 目錄樹都按字母排，數字前綴是免費的閱讀指引
- **「依角色找文件」表是必備**：README 應該提供「你是主任 → 看 X、你是 IT → 看 Y、你是講師 → 看 Z」的角色導覽，讓不同讀者各取所需

## 決策 3（QODA 第三輪）：執行節奏

**情境**：全套重做 admin-project-workflow（README + docs + Apps Script + 教材）約 12 小時實際工時。一次跑完還是分階段？

**選項**：
- A. **一次跑完**：12 小時連續工作，最後一次交付
- B. **分四階段交付**（P1 骨架 / P2 文件 / P3 程式 / P4 教材＋範例），每階段使用者檢查再進下一步
- C. **只做最小可用版（P1 骨架）後就 push**，其他陸續更新

**決定**：選 B。

**考量**：
1. A 風險高：12 小時跑完才檢查，若 P1 骨架不對，後面 P2/P3/P4 全部白做
2. **B 每階段 30 分鐘到 2 小時，使用者可即時驗證方向，最壞情況只浪費當前階段**
3. C 「先 push 再說」對 GitHub 公開印象不好——首版內容不完整會讓潛在使用者跳走
4. B 對應 user_preferences 的「3+ 步驟要先 plan、非小事決策要 QODA」紀律

**教訓**：
- **分階段交付的本質是「決策點頻率」**：每階段結束 = 一個 checkpoint，使用者能介入、能調整，比一次大爆炸更穩
- **「最小可用」（C 方案）的迷思**：對工具型 repo，最小可用版本反而會讓首批使用者覺得「不夠成熟」而流失。完整版上線後再來推廣 ROI 更高

## 反思

**做得好的**：
- **QODA 三輪都明確 sign-off 才動手**：方案 B → 目錄結構 D → 分四階段，每輪都有書面 approval
- **每輪 Options 都列 ≥ 2 個 + 標 ★ 推薦**：對應 user_preferences QODA 協定的第 2 步
- **Decision 一句話 + 為什麼**：對應 QODA 第 3 步「明示推薦哪個 + 為什麼」
- **Approval 等使用者「OK」才動手**：QODA 第 4 步嚴格執行

**可以更好的**：
- **QODA 過程當下沒寫成決策日誌**：Phase 2.5 才補寫，違反「即時寫」原則
- **三輪 QODA 各自獨立呈現**：實際上「結構 / 目錄 / 執行節奏」三個決策互相影響，應該作為單一 plan 提交而非分三輪
- **沒列出「使用者可能拒絕」的反例 scenarios**：例如若使用者選 A 雙主線，我該怎麼回應？預先想好可以加快第二輪對話

**對長期專案的影響**：
- **QODA 協定第一次在實戰被完整應用**：personal-playbook §5.9 從「prompt 段落」變成「真實 workflow 紀錄」
- **未來其他 fork / 新 repo 都可參考本檔的 QODA 三輪模板**
- **「方案 B 聚焦」的決定影響後續所有 P1-P4 的內容方向**：README 痛點 hook、docs 角色導覽、src/ 模組拆分都依「主任為主、IT 為輔」這條準繩展開
